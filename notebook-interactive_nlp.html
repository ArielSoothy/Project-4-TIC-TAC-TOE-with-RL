1.2Why Precision & Recall Beat Accuracy

❓ What we needed to explain:

Why precision and recall are better metrics than accuracy for this imbalanced dataset.

📈 Our analysis:

Accuracy Deception

Recall True Positives / (TP + False Negatives) 0% - misses all emotions Shows coverage of actual emotions

💡 Key Learning

A model that always predicts "neutral" gets 82% accuracy but 0% precision and recall. This shows accuracy can be completely misleading with imbalanced data!

🤔 What it means

In simple terms: Imagine a talent scout at a music competition who needs to find amazing singers. If they just say "no talented singers here" at every show, they'll be "right" 90% of the time (since most people aren't professional-level). But they'll miss every single star! 🌟 Accuracy says "90% correct predictions" but precision and recall reveal the truth: "0% actual talent discovered." That's why we need smarter ways to measure success - we want to find those hidden gems! 💎

🔍 Key Terms for This Question

Accuracy

What it is: Simple percentage of correct predictions out of total predictions.

Formula: (Correct Predictions) / (Total Predictions)

The trap: Looks great with imbalanced data but can be completely misleading!

Example: Predicting "no cancer" for everyone gives 95% accuracy if only 5% have cancer, but misses all actual cases! 🚨

Precision

What it is: Of all the positive predictions you made, how many were actually correct?

Formula: True Positives / (True Positives + False Positives)

Think of it as: Quality control - "When I say YES, how often am I right?"

High precision: You rarely cry wolf, but you might miss some real cases. 🎯

Recall

What it is: Of all the actual positive cases, how many did you successfully find?

Formula: True Positives / (True Positives + False Negatives)

Think of it as: Coverage - "Of all the real cases, how many did I catch?"

High recall: You catch almost everything, but might have some false alarms. 🔍

2.3Custom ROUGE Implementation & Analysis

🔧 What we implemented:

Built ROUGE-N evaluation from scratch (no external libraries).

ROUGE Implementation
Python 
def calculate_rouge_n(generated, reference, n=1):
    """Custom ROUGE-N implementation"""
    generated_ngrams = set(nltk.ngrams(generated.split(), n))
    reference_ngrams = set(nltk.ngrams(reference.split(), n))
    
    if len(reference_ngrams) == 0:
        return 0.0
    
    overlap = generated_ngrams.intersection(reference_ngrams)
    return len(overlap) / len(reference_ngrams)

It's like comparing a book to its movie adaptation: same story, totally different presentation! 📚➡️🎬

🔍 Key Terms for This Question

ROUGE (Recall-Oriented Understudy for Gisting Evaluation)

What it is: A scoring system that measures how much overlap exists between a generated summary and reference summaries.

ROUGE-1: Counts individual word matches. Higher = more words in common.

ROUGE-2: Counts word-pair (bigram) matches. More strict than ROUGE-1.

Why for summarization: Unlike BLEU (translation), ROUGE focuses on recall - catching important information rather than perfect word order.

Limitation: Misses paraphrasing! "Happy" vs "joyful" = 0% similarity even though they mean the same. 😊

N-grams

What it is: Sequences of N consecutive words from text.

1-gram (unigram): Single words - "the", "cat", "sat"

2-gram (bigram): Word pairs - "the cat", "cat sat"

3-gram (trigram): Three words - "the cat sat"

Why useful: Captures word order and context. "New York" (bigram) means something different than "New" + "York" separately! 🗽

Abstractive vs Extractive Summarization

Extractive: Copy-paste approach - selects important sentences from original text.

Abstractive: Creative rewriting - generates new sentences that capture the meaning.

Analogy: Extractive = highlighting key sentences; Abstractive = writing your own summary after reading.

ROUGE challenge: Works great for extractive, struggles with abstractive (different words, same meaning). 📝

    """Find most semantically similar article using embeddings"""
    most_relevant_article = None
    max_similarity = -1

🔍 Key Terms for This Question

Semantic Search

What it is: Search that understands meaning, not just keyword matching.

vs Keyword search: Keyword finds "dog"; Semantic finds "dog", "puppy", "canine", "pet" for the same query.

How it works: Converts text to mathematical representations (embeddings) that capture meaning.

Real examples: Google understands "weather tomorrow" = "forecast", Amazon knows "phone charger" = "mobile cable". 🔍

Power: Handles typos, synonyms, different languages, and conceptual similarity!

Embeddings (Vector Representations)

What they are: Mathematical "fingerprints" that capture the meaning of words/sentences as numbers.

How they work: Similar meanings get similar numbers. "King" and "Queen" have close embeddings.

Dimensions: Our DistilBERT creates 768-dimensional vectors (768 numbers per text piece).

Magic property: Math with meaning! "King" - "Man" + "Woman" ≈ "Queen" in embedding space! 👑

Why powerful: Computers can now "understand" text by doing math on these meaning representations. 🧮✨

Cosine Similarity

What it is: A way to measure how similar two vectors (embeddings) are, ranging from -1 to 1.

How it works: Measures the angle between vectors - small angle = very similar, large angle = different.

vs Euclidean distance: Cosine focuses on direction (meaning), not magnitude. Perfect for high-dimensional embeddings!

Score meaning: 1.0 = identical, 0.9+ = very similar, 0.5 = somewhat related, 0.0 = unrelated, -1.0 = opposite. 📐

Why we use it: Works great even when embeddings have different "lengths" - focuses purely on meaning similarity! 🎯

🔍 Key Terms for This Question

Apache Spark

What it is: Distributed computing framework for processing massive datasets across multiple machines.

Why for embeddings: Can generate embeddings for millions of documents in parallel using GPU clusters.

Speed: Instead of processing 1 document at a time, process 1,000+ simultaneously across different machines.

Real-world use: Netflix uses Spark to process viewing data, Uber for ride analytics. Enterprise-grade! 🚀

Vector Databases (Pinecone/FAISS)

What they are: Specialized databases optimized for storing and searching high-dimensional vectors (embeddings).

FAISS: Facebook's library for similarity search - can handle billions of vectors with millisecond search times.

Pinecone: Cloud-based vector database with auto-scaling and real-time updates.

vs Regular databases: Regular DBs store text/numbers, vector DBs store meaning and can search by similarity! 🔍

Scale: Can store and search billions of documents - like having Google's search capability! 🌐

HNSW (Hierarchical Navigable Small World)

What it is: Super-efficient algorithm for finding similar vectors in high-dimensional space.

How it works: Creates a multi-layer graph structure - like having express highways AND local roads for navigation.

Speed magic: Instead of checking every single vector, it "hops" through the graph to find similar ones quickly.

Performance: Can search through millions of vectors in under 1 millisecond! ⚡

Real-world: Powers search in Spotify (find similar songs), Pinterest (similar images), and many AI applications. 🎵📸

FastAPI + Kubernetes

FastAPI: Modern Python framework for building super-fast APIs. Auto-generates documentation and handles async requests.

Kubernetes: Container orchestration system that automatically manages, scales, and heals applications.

Together: FastAPI handles the search requests, Kubernetes ensures the system can handle thousands of simultaneous users.

Auto-scaling: If traffic increases, Kubernetes automatically spins up more servers. If it decreases, it scales down to save money! 💰

Production ready: Used by companies like Netflix, Airbnb, and Google for handling massive traffic. 🌐
